# autoresearch

这是一个让大语言模型(LLM)自主进行研究的实验。

## 准备工作

在开始一个新实验时,与用户协作完成以下步骤:

1. **约定运行标签(run tag)**:基于今天的日期提议一个标签(例如 `mar5`)。分支 `autoresearch/<tag>` 必须不存在 —— 这是一次全新的运行。
2. **创建分支**:从当前的 master 分支执行 `git checkout -b autoresearch/<tag>`。
3. **阅读相关文件**:本仓库较小,请阅读以下文件以获取完整上下文:
   - `README.md` —— 仓库背景说明。
   - `prepare.py` —— 固定常量、数据准备、分词器(tokenizer)、数据加载器(dataloader)、评估逻辑。请勿修改。
   - `train.py` —— 你要修改的文件。包含模型架构、优化器、训练循环。
4. **确认数据存在**:检查 `~/.cache/autoresearch/` 是否包含数据分片(data shards)和分词器。如果没有,请告知用户运行 `uv run prepare.py`。
5. **初始化 results.tsv**:创建一个只包含表头行的 `results.tsv`。基线(baseline)将在第一次运行后记录。
6. **确认并开始**:确认准备工作无误。

获得确认后,即可启动实验。

## 实验过程

每个实验运行在单个 GPU 上。训练脚本以**固定 5 分钟的时间预算**(墙钟训练时间,不包括启动/编译时间)运行。你只需执行:`uv run train.py`。

**你可以做的事:**
- 修改 `train.py` —— 这是你唯一可以编辑的文件。一切都可以更改:模型架构、优化器、超参数、训练循环、批量大小(batch size)、模型规模等。

**你不能做的事:**
- 修改 `prepare.py`。它是只读的。它包含固定的评估逻辑、数据加载、分词器和训练常量(时间预算、序列长度等)。
- 安装新包或添加依赖项。只能使用 `pyproject.toml` 中已有的内容。
- 修改评估框架(evaluation harness)。`prepare.py` 中的 `evaluate_bpb` 函数是基准真相指标(ground truth metric)。

**目标很简单:获得最低的 val_bpb。** 由于时间预算是固定的,你无需担心训练时间 —— 它总是 5 分钟。一切都可以更改:架构、优化器、超参数、批量大小、模型规模。唯一的约束是代码必须能在不崩溃的情况下运行,并在时间预算内完成。

**显存(VRAM)** 是软约束。为了获得有意义的 val_bpb 提升,显存的适度增加是可以接受的,但不应该急剧膨胀。

**简洁性标准**:在其他条件相同的情况下,越简洁越好。一个引入丑陋复杂性的小改进是不值得的。反过来,删除某些内容却获得了相等或更好的结果,这是一个绝佳的结果 —— 这是简化的胜利。在评估是否保留一个改动时,要权衡复杂性成本和改进幅度。一个仅提升 0.001 val_bpb 却增加了 20 行 hacky 代码的改动?可能不值得。一个通过删除代码就能提升 0.001 val_bpb 的改动?绝对要保留。改进约为零但代码变得更简洁?保留。

**第一次运行**:你的第一次运行应当始终是建立基线,因此保持训练脚本原样运行即可。

## 输出格式

脚本完成后,会打印类似如下的摘要:

```
---
val_bpb:          0.997900
training_seconds: 300.1
total_seconds:    325.9
peak_vram_mb:     45060.2
mfu_percent:      39.80
total_tokens_M:   499.6
num_steps:        953
num_params_M:     50.3
depth:            8
```

注意脚本被配置为始终在 5 分钟后停止,因此根据计算平台的不同,数字可能会有差异。你可以从日志文件中提取关键指标:

```
grep "^val_bpb:" run.log
```

## 记录结果

实验完成后,将结果记录到 `results.tsv`(制表符分隔,**不是**逗号分隔 —— 因为描述中可能出现逗号,会导致解析错误)。

该 TSV 文件有一个表头行和 5 列:

```
commit	val_bpb	memory_gb	status	description
```

1. git commit 哈希(短格式,7 个字符)
2. 取得的 val_bpb(例如 1.234567)—— 崩溃时记为 0.000000
3. 峰值显存,单位为 GB,保留一位小数(例如 12.3 —— 即 peak_vram_mb 除以 1024)—— 崩溃时记为 0.0
4. 状态:`keep`(保留)、`discard`(丢弃)或 `crash`(崩溃)
5. 该实验尝试内容的简短文字描述

示例:

```
commit	val_bpb	memory_gb	status	description
a1b2c3d	0.997900	44.0	keep	baseline
b2c3d4e	0.993200	44.2	keep	increase LR to 0.04
c3d4e5f	1.005000	44.0	discard	switch to GeLU activation
d4e5f6g	0.000000	0.0	crash	double model width (OOM)
```

## 实验循环

实验运行在一个专用分支上(例如 `autoresearch/mar5` 或 `autoresearch/mar5-gpu0`)。

无限循环:

1. 查看 git 状态:当前所在的分支/提交。
2. 通过直接修改代码,在 `train.py` 中尝试一个实验性想法。
3. git commit。
4. 运行实验:`uv run train.py > run.log 2>&1`(重定向所有输出 —— **不要**使用 tee,也不要让输出淹没你的上下文)。
5. 读取结果:`grep "^val_bpb:\|^peak_vram_mb:" run.log`。
6. 如果 grep 输出为空,说明运行崩溃了。运行 `tail -n 50 run.log` 读取 Python 栈追踪信息并尝试修复。如果经过几次尝试仍无法成功,就放弃。
7. 将结果记录到 tsv 文件中(注意:不要将 results.tsv 文件提交到 git,让它保持未跟踪状态)。
8. 如果 val_bpb 改善了(更低),你就"推进"分支,保留 git commit。
9. 如果 val_bpb 持平或变差,git reset 回到起点。

理念是你作为一个完全自主的研究者在尝试各种想法。有效就保留,无效就丢弃。同时通过推进分支来迭代。如果你感觉陷入了某种困境,可以回退,但应当极少这样做(如果有的话)。

**超时**:每个实验应耗时约 5 分钟(加上几秒的启动和评估开销)。如果一次运行超过 10 分钟,杀掉它并视为失败(丢弃并回退)。

**崩溃**:如果一次运行崩溃了(OOM、bug 等),用你的判断:如果是一些容易修复的小问题(例如拼写错误、缺失导入),就修复并重新运行。如果想法本身根本不可行,直接跳过它,在 tsv 中记为"crash"状态,然后继续。

**绝不停止**:一旦实验循环开始(在初始设置之后),**不要**停下来询问用户是否应该继续。**不要**问"我应该继续吗?"或"这是一个好的停止点吗?"。用户可能在睡觉,或者离开了电脑,期望你**无限期**地持续工作,直到被手动停止。你是自主的。如果你想不出新点子,就更努力地思考 —— 阅读代码中引用的论文,重新阅读相关文件以寻找新角度,尝试组合之前的近似成功案例,尝试更激进的架构变化。循环一直运行,直到用户中断,就这么简单。

举个例子,用户可能在睡觉时让你持续运行。如果每次实验耗时约 5 分钟,那么你每小时可以运行约 12 次,在普通人睡眠时间内总共约 100 次。用户醒来时就能看到你在他们睡觉期间完成的所有实验结果!
